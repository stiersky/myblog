🎯 安全困境

AI 助手越来越强大，但数据安全始终是悬在头顶的达摩克利斯之剑。尤其是处理标书、报价、客户信息这些敏感数据时，每一个决定都可能带来风险。

我面临的选择题：

  • 用云端大模型（GPT-4、Claude）？能力强，但数据要上传
  • 用本地模型（Ollama）？数据不出域，但性能有限
  • 混合使用？如何划定安全边界？

📊 本地模型 vs 云端模型对比

维度 本地模型 (Ollama) 云端模型 (GPT-4/Claude)
数据安全性 高 - 数据不出本地 低 - 数据上传第三方
模型能力 中 - Qwen3.5 等开源模型 高 - 最新闭源模型
响应速度 快 - 无网络延迟 中 - 依赖网络
成本 一次性硬件投入 按 Token 计费
可用性 离线可用 依赖网络和服务
隐私合规 完全可控 依赖服务商政策
💡 结论：没有绝对的好坏，关键是根据数据敏感度选择合适的模型。

🔧 Ollama 本地部署实战

我选择在 MacBook 上部署 Ollama，运行 Qwen3.5 模型。下面是完整配置过程。

1. 安装 Ollama

# macOS 安装 brew install ollama # 启动服务 ollama serve # 验证安装 ollama list

2. 下载模型

# 下载 Qwen3.5（推荐 7B 或 14B 版本） ollama pull qwen3.5 # 查看已下载模型 ollama list # 输出示例： # NAME ID SIZE MODIFIED # qwen3.5:7b a1b2c3d4e5f6 4.7 GB 2 days ago

3. 配置 OpenClaw 使用本地模型

# ~/.openclaw/config/models.yaml models: default: ollama/myqwen3.5 providers: ollama: baseUrl: http://127.0.0.1:11434 models: - id: myqwen3.5 name: Qwen 3.5 (Local) contextLength: 32768 temperature: 0.7
✅ 关键配置：baseUrl 设置为 127.0.0.1，确保只监听本地回环接口，不暴露到外网。

4. 性能优化

MacBook 内存有限（17GB），需要优化 Ollama 的资源使用：

# 限制 Ollama 使用的 GPU 显存（如适用） export OLLAMA_NUM_GPU=1 # 限制并发请求数 export OLLAMA_MAX_QUEUE=2 # 设置模型卸载策略（内存不足时自动卸载） export OLLAMA_KEEP_ALIVE=5m

🛡️ 敏感数据分类策略

我制定了明确的数据分类标准，决定哪些数据可以用云端模型，哪些必须本地处理。

🔴 红色级别（必须本地处理）

  • 标书内容（含技术方案、报价）
  • 客户名单和联系方式
  • 内部财务数据
  • 未公开的产品规划
  • 认证凭证（API Key、密码）

🟡 黄色级别（可脱敏后云端处理）

  • 公开的市场分析报告
  • 已发布的产品文档
  • 通用的技术方案模板
  • 行业趋势分析

🟢 绿色级别（可自由使用云端模型）

  • 公开的新闻文章
  • 技术博客草稿
  • 代码片段（不含业务逻辑）
  • 学习材料

🚧 云端 API 安全边界

即使使用云端模型，也要建立安全边界，降低风险。

1. API Key 管理

# ❌ 错误做法：硬编码在代码中 const apiKey = "sk-xxxxxxxxxxxxx"; # ✅ 正确做法：使用环境变量 const apiKey = process.env.OPENAI_API_KEY; # ✅ 更好：使用 macOS Keychain security find-generic-password -s "openai-api-key" -w

2. 请求审计

记录所有云端 API 调用，便于事后审计：

# OpenClaw 配置启用日志 logging: apiCalls: true redactSensitiveData: true retention: 30d

3. 数据脱敏

发送给云端模型前，自动脱敏敏感信息：

💡 脱敏策略：
  • 客户名称 → [客户 A]
  • 金额数字 → [金额]
  • 日期 → [日期]
  • 地址 → [地址]

⚠️ 真实安全事件学习

🚨 2026-03-14 CNCERT 安全预警

问题：Python HTTP Server 绑定到所有接口，导致临时服务暴露到公网。

# ❌ 危险！全网可访问 python -m http.server 8080 # ✅ 安全！仅本地访问 python -m http.server 8080 --bind 127.0.0.1

教训总结

  • 默认绑定 127.0.0.1：任何服务默认只监听本地，除非确需外部访问
  • 临时隧道及时关闭：ngrok、localtunnel 用完立即关闭
  • 定期检查暴露端口：netstat -an | grep LISTEN
  • 不安装来路不明的 Skill：只使用可信来源的扩展

🔐 混合使用策略

实际工作中，我采用混合策略：本地模型处理敏感任务，云端模型处理通用任务。

场景 1：编写标书

模型选择：本地 Ollama + Qwen3.5

原因：标书含敏感技术方案和报价，不能上传云端

场景 2：写技术博客

模型选择：云端 GPT-4 / Claude

原因：博客内容是公开分享，无敏感信息

场景 3：代码审查

模型选择：根据代码类型决定

业务代码：本地模型

开源组件：云端模型

场景 4：市场分析

模型选择：云端模型（脱敏后）

原因：使用公开数据，脱敏客户名称后即可

📋 安全检查清单

每次使用 AI 助手前，我都会过一遍这个清单：

✅ 数据安全检查

  • [ ] 这份数据是否包含敏感信息？
  • [ ] 如果敏感，是否已脱敏？
  • [ ] 选择的模型是否符合安全级别？
  • [ ] API Key 是否安全存储？
  • [ ] 是否有审计日志？

✅ 系统安全检查

  • [ ] Ollama 是否只绑定 127.0.0.1？
  • [ ] 是否有不必要的端口暴露？
  • [ ] 临时隧道是否已关闭？
  • [ ] 文件权限是否正确（chmod 600）？

📈 性能对比实测

我在 MacBook 上实测了本地模型和云端模型的性能差异：

任务类型 本地 Qwen3.5 云端 GPT-4
邮件回复（200 字） ~3 秒 ~5 秒
代码生成（50 行） ~15 秒 ~8 秒
文档总结（5000 字） ~30 秒 ~12 秒
复杂推理 能力有限 表现优秀
💡 实测结论：对于日常任务（邮件、简单代码、文档处理），本地模型完全够用。复杂推理和创意任务，云端模型仍有优势。

💬 结语

安全不是二选一，而是根据场景做权衡。我的策略很明确：

敏感数据本地化，通用任务云端化，边界清晰可审计。

Ollama 等本地模型的崛起，让我们有了更多选择。不必在安全和能力之间做痛苦抉择，可以兼得。

当然，本地模型的能力还在快速进化。也许不久的将来，本地模型就能完全替代云端模型处理所有任务。但在那之前，混合策略是最务实的选择。🦞
