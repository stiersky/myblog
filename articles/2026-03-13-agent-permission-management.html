给 AI 助手的权限，应该像给员工的权限——够用就好，不能过度

📅 2026 年 3 月 13 日 🔐 权限管理
💡 核心观点：AI Agent 的权限管理不是可选项，是必选项。

🎯 为什么需要权限管理

CNCERT 安全预警发布后，我深刻意识到一个问题：我给大总管的权限太高了。

想想看，如果大总管被攻击者控制，会发生什么：

  • 用我的飞书账号向同事发送"紧急消息"
  • 读取硬盘里的所有合同、财务文件
  • 拿走 ~/.ssh/ 里的密钥，登录所有服务器
  • 以我的身份调用公司内部系统

这不只是数据泄露，是数字身份被盗用。

🔐 三级权限体系

我设计了一个三级权限体系：

L1：只读权限（安全）✅

  • 读取工作区文件
  • 搜索记忆库
  • 查看系统状态
  • 读取公开 API 数据

适用场景：日常问答、信息查询、数据分析

L2：有限写入（中等风险）⚠️

  • 写入工作区文件
  • 发送消息到指定频道
  • 调用已授权的 API
  • 创建/编辑文档

适用场景：文档编写、消息发送、任务执行
安全要求：需要人工审核关键操作

L3：系统权限（高风险）🔴

  • 执行终端命令
  • 安装/卸载软件
  • 修改系统配置
  • 访问敏感文件（SSH 密钥、密码）

适用场景：系统维护、紧急修复
安全要求：必须人工确认，最好物理隔离

🛠️ 实践方案

1. 本地模型优先

能用本地模型就不用云端模型。本地模型不会把数据传到外部，天然安全。

"models": {
"defaults": {
"primary": "ollama/myqwen3.5", // 本地模型
"fallbacks": ["qwencode/qwen3.5-plus"] // 云端备用
}
}

2. 敏感数据隔离

SSH 密钥、API 密钥、密码等敏感数据，不应该放在 Agent 能访问的地方。

// ❌ 危险：Agent 工作区包含敏感文件
~/.openclaw/workspace/
├── .env # 包含 API 密钥
├── config.json # 包含密码
└── .ssh/ # SSH 密钥

// ✅ 安全：敏感数据独立存储
~/.openclaw/secure/ # 加密存储，Agent 无法访问
~/.openclaw/workspace/ # 普通工作区

3. 命令白名单

禁止 Agent 执行任意命令，只允许白名单中的命令：

"commands": {
"bash": false, // 禁止任意 bash 命令
"native": true, // 允许内置安全命令
"useAccessGroups": true // 使用访问组控制
}

⚠️ 安全教训

教训 1：Python HTTP Server 暴露

为了测试飞书 webhook，我运行了：

# ❌ 危险！全网可访问
python -m http.server 8080

# ✅ 安全！仅本地访问
python -m http.server 8080 --bind 127.0.0.1

这个错误让我惊出一身冷汗。如果当时被扫描到，整个工作区文件都会泄露。

教训 2：临时隧道忘记关闭

为了飞书机器人接收消息，我开了 localtunnel 隧道。配置完成后忘记关闭，导致 webhook 端点一直暴露在公网。

改进：临时隧道用完立即关闭，或者用脚本自动清理。

🎯 最佳实践总结

  1. 默认拒绝：除非明确需要，否则不给权限
  2. 最小权限：只给完成任务所需的最小权限
  3. 分层隔离：敏感操作需要额外确认
  4. 定期审计：每周检查一次权限配置
  5. 物理隔离：最安全的方案是专用机器
权力越大，责任越大。给 AI 的权限也一样。
