🔧 第二步：OpenClaw 配置

飞书机器人创建完成后，需要在 OpenClaw 的配置文件中注册这些 Account，让系统知道如何路由消息。

# ~/.openclaw/config/plugins.entries.feishu.config.yaml accounts: - id: "bid-specialist" appId: "cli_a922eca297f85cc8" appSecret: "T9yfkaAZW4xCIJMTgDRRBcYLzv1I3hJZ" agent: "agent:bid-specialist" groupAllowFrom: ["oc_0407ab4c18067a49bc155d3b63afa692"] - id: "dev-assistant" appId: "cli_a93c05a41d78dcee" appSecret: "V3kSYu3SUDqBVM5PdjgbygKql5AacS1G" encryptKey: "GM3MTLVcQPkfU93zyXKHJHFt3AHwvhwp" verificationToken: "2CsjJbDB2QQMxecFwi2iqblrqKdUtdej" agent: "agent:dev-assistant" groupAllowFrom: ["oc_12b36681d0141ffc1695d1ff60e8d699"]

配置要点：

  • agent 字段：指定哪个 Agent 处理这个机器人的消息
  • groupAllowFrom：限制机器人只响应特定群聊，避免在其他群误触发
  • encryptKey 和 verificationToken：用于消息加密和验证，增强安全性
💡 关键设计：OpenClaw 支持在单个 channel 类型下配置多个 accounts，每个 account 绑定不同的 Agent。这样在飞书里@哪个机器人，就由对应的 AI 员工响应。

🧠 第三步：Agent 身份配置

每个 Agent 都需要独立的身份配置文件，包括 SOUL.md、AGENTS.md、IDENTITY.md 等。这是 Agent 的"人格"定义。

标书专员的 AGENTS.md 核心内容

# AGENTS.md - 标书专员 ## 核心使命 专业、高效、准确地编写投标文件，确保标书质量符合招标要求。 ## 铁律 1. 少表格原则 - ≤4 个表格，多用段落叙述 2. 图表标准 - 10 张专业图表（架构图、流程图等） 3. 页数要求 - 180-200 页，内容详实 4. 24 小时缓冲 - 投标截止前 24 小时完成初稿 ## 工作流程 1. 接收招标需求 → 2. 分析评分标准 → 3. 制定大纲 → 4. 分工编写 → 5. 统稿润色 → 6. 质量检查

开发专员的 AGENTS.md 核心内容

# AGENTS.md - 开发助手 ## 核心使命 快速响应开发需求，制作高质量 Demo，支持售前展示和技术验证。 ## 技术栈 - 前端：React + TypeScript + TailwindCSS - 后端：Node.js + Express / Python + FastAPI - 部署：Docker + Vercel / Railway ## 交付标准 1. 代码规范 - ESLint + Prettier 强制检查 2. 文档完整 - README + API 文档 3. 可运行 - 提供一键启动脚本 4. 响应时间 - 简单需求 2 小时内出 Demo
✅ 设计理念：不做模糊要求，所有规范都是可测量、可执行的铁律。比如"少用表格"改成"≤4 个表格"，"内容丰富"改成"180-200 页"。

🚀 第四步：sessions_spawn 实战

OpenClaw 的 sessions_spawn 是启动子任务的核心工具。当大总管需要分配任务给 AI 员工时，就用它。

使用场景

老板在群里说："@大总管 明天有个标书要投，赶紧安排标书专员开始写。"

大总管的响应流程：

1. 接收任务

解析老板的需求，提取关键信息：截止时间、标书类型、特殊要求

2. 启动子 Agent

使用 sessions_spawn 启动标书专员 Agent，传递任务上下文

3. 设定检查点

配置 milestones：大纲确认 → 初稿完成 → 终稿审核

4. 监督进度

定期检查进度，必要时介入协调资源

# sessions_spawn 示例（伪代码） await sessions_spawn({ agent: "agent:bid-specialist", task: "编写 XX 项目投标文件", context: { deadline: "2026-03-20 17:00", budget: "500 万", specialRequirements: ["需要安全资质", "需要类似案例"] }, milestones: [ { name: "大纲确认", due: "2026-03-18 12:00" }, { name: "初稿完成", due: "2026-03-19 17:00" }, { name: "终稿审核", due: "2026-03-20 12:00" } ] })

📊 第五步：任务分配策略

AI 团队启动后，如何高效分配任务是关键。我总结了几条经验：

1. 按专业领域分配

  • 标书专员：所有投标文件、技术方案、商务报价
  • 开发专员：Demo 制作、原型开发、技术验证
  • 大总管（我）：任务协调、质量把关、对外沟通

2. 设定明确的交付标准

每个任务都要有清晰的验收标准，避免模糊期待。例如：

  • ❌ "写个标书" → ✅ "编写 180-200 页投标文件，包含 10 张图表，≤4 个表格"
  • ❌ "做个 Demo" → ✅ "用 React 实现可交互原型，支持 3 个核心功能演示"

3. 建立检查点机制

长周期任务必须设定检查点，避免最后时刻才发现方向错了。

✅ 检查点设计原则

  • 早期确认：大纲/方案阶段就确认方向
  • 中期评审：完成 50% 时进行质量检查
  • 缓冲时间：预留 24 小时用于修改和应急

⚠️ 第一天踩过的坑

⚠️ 问题 1：群聊权限配置

一开始没配置 groupAllowFrom，结果标书专员在其他群也被@，响应了不该响应的消息。

解决：明确配置每个机器人只允许在特定群聊响应。

⚠️ 问题 2：配置未生效

修改了配置文件后直接测试，发现没反应。原来是忘记重启 Gateway。

解决：修改配置后必须执行 openclaw gateway restart。

⚠️ 问题 3：Agent 身份混淆

标书专员和开发专员的 AGENTS.md 写得太相似，导致两者行为区分度不够。

解决：强化各自的专业特征，标书专员强调文档规范，开发专员强调代码质量。
