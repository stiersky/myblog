# 打造 AI 算力永动机：OpenClaw + CLIProxyAPI 实战

最近测试了一套有意思的方案：用 OpenClaw 搭配 CLIProxyAPI，搞了个"算力永动机"。

先说结论：**这玩意儿真香，但可能不是你想象的那样。**

## 什么是算力永动机？

简单说，就是把多个 AI 账号的额度聚合起来，自动轮询、容灾降级、额度叠加。听起来很美好，对吧？

我实际测试了一下咱们现有的配置：

- **Doubao（火山引擎）**：7 个模型，免费额度
- **Qwencode（阿里云/硅基流动）**：8 个模型，包括 Qwen3.5 Plus、Kimi K2.5、GLM-5
- **Ollama 本地**：qwen3.5:9b，零成本兜底

总共 16 个模型，已经天然具备了"永动机"的雏形。

## CLIProxyAPI 能带来什么？

CLIProxyAPI 是个万能网关，能把各种非标准 AI 接口统一成 OpenAI 规范。它的核心价值：

1. **多账号聚合** - 多个 Gemini/Kimi 账号合并使用
2. **自动轮询** - 单账号限流时自动切换
3. **容灾降级** - 付费额度用完自动切免费模型
4. **额度叠加** - 多个免费账号变"无限"额度

## 但等等，咱们需要吗？

测试完我有个反直觉的发现：**对于大多数个人用户，CLIProxyAPI 可能不是必须的。**

原因很简单：

- 咱们已经有 Doubao + Qwencode 双 provider，天然容灾
- 16 个模型覆盖通用、代码、视觉场景，够用
- 本地 Ollama 兜底，零成本、隐私安全
- 配置简洁，直接 OpenAI 兼容接口

CLIProxyAPI 真正适合的场景是：
- 你有多个同平台账号（比如 5 个 Gemini 账号）
- 需要聚合不同平台的免费额度
- 想要更细粒度的轮询和降级策略

## 我的建议

**先优化现有配置，再考虑扩展。**

具体做法：

1. **配置智能降级策略**
   ```
   Qwen3.5 Plus (主) → Qwen3 Max → Kimi K2.5 → GLM-5 → 本地 Ollama (兜底)
   ```

2. **监控额度和使用情况**
   - 定期检查各平台剩余额度
   - 设置用量告警

3. **按需引入 CLIProxyAPI**
   - 如果现有额度不够用
   - 如果有多个同平台账号需要聚合

## 最后的思考

"算力永动机"这个概念很吸引人，但本质上是**资源管理优化**，不是魔法。

对于个人用户，更重要的是：
- 清楚自己需要什么模型
- 合理配置降级策略
- 该花钱时花钱，该省钱时省钱

技术是工具，不是目的。能用现有配置解决问题，就别折腾新方案。

当然，如果你有更多账号资源想聚合，CLIProxyAPI 确实是个好选择。只是别被"永动机"这个名字忽悠了——它只是个聪明的调度器，不是无限能源。
